## Design Context

### Users
Jarvis users are people setting up and operating a small private assistant fleet
across Macs and Raspberry Pis. They may be technical enough to own the hardware,
but the install/update experience should not require developer habits, repo
clones, manual `.env` editing, or knowledge of `uv`.

### Brand Personality
Jarvis should feel calm, capable, and exact. The product should give the user
confidence that every machine is known, paired, updateable, and recoverable.

### Aesthetic Direction
Use a refined appliance-control aesthetic: bright, precise, tactile, and
operational. Avoid generic SaaS dashboards, dark neon developer styling, and
command-heavy setup pages. The public site should make installation feel guided
and trustworthy rather than experimental.

### Design Principles
- Lead with the actual install path, not marketing copy.
- Show the fleet as physical roles: brain, workers, intercoms, Pi rooms.
- Prefer clear state and next actions over explanations.
- Treat security as visible reassurance: pairing, private network, no secrets in git.
- Keep developer details available but secondary.
